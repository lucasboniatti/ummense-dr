## Story

As a **user debugging automation failures**, I want to **see detailed execution history with full context (inputs, outputs, errors, timings)**, so that **I can understand what went wrong and fix it**.

## Acceptance Criteria

- [x] **[CRITICAL]** Execution history table shows all automation executions with key metrics
  - Columns: Execution ID, Automation name, Timestamp, Status (success/failed), Duration, Trigger type
  - Sortable: By timestamp (default), status, duration
  - Paginated: 50 rows per page
  - Filters: Automation, date range (last 24h, 7d, 30d, custom), status
  - Search: By execution ID or automation name
- [x] **[HIGH]** Execution detail page shows step-by-step breakdown with inputs/outputs
  - Step timeline view: Vertical timeline showing each step in execution order
  - Per-step details: Step name, status, input data, output data, duration
  - Expandable: Click step to see full JSON input/output
  - Visual indicators: Success (green), Failed (red), Skipped (grey)
- [x] **[HIGH]** Error details include exception message, stack trace, and state snapshot
  - Error panel on detail page: Exception message (large text)
  - Stack trace: Multi-line code reference (file, line number)
  - State snapshot: JSON object showing execution state when error occurred
  - Reproducible: All data needed to understand and reproduce the error
- [x] **[HIGH]** Audit log captures all user actions (create, modify, enable, disable, manual retry)
  - Audit table: user_id, action, automation_id, changes, timestamp
  - Action types: create_automation, modify_automation, enable_schedule, disable_schedule, manual_retry, delete_automation
  - Changes: What was changed (e.g., {old: "0 9 * * *", new: "0 10 * * *"})
  - User attribution: Audit log shows who made each change
- [x] **[x]** Execution history searchable by automation, date range, status, error type
  - Advanced search: Combine filters (automation AND date range AND status)
  - Error type filter: Show only failures of specific error type (e.g., timeout, 404, validation)
  - Search terms: Partial match on automation name, execution ID
  - Recent searches: Save/suggest previous searches
- [x] **[MEDIUM]** Exports supported (CSV, JSON) for further analysis
  - CSV export: Include all searchable fields + error message
  - JSON export: Full execution context for detailed analysis
  - Limit: Max 50K rows per export (prevents memory issues)
  - Background job: Large exports return later via email
- [x] **[MEDIUM]** Retention policy: 90 days default, configurable up to 2 years
  - User setting: Retention days (default 90, max 730)
  - Automated cleanup: Nightly job deletes records >retention_days old
  - Archival option: Export to cold storage (S3) before deletion
  - Compliance: Audit logs immutable (no editing/deletion)

### Database Schema

```sql
-- Table: automation_executions (extended from Story 3.1)
CREATE TABLE automation_executions (
  id UUID PRIMARY KEY,
  automation_id UUID NOT NULL REFERENCES automations(id),
  user_id UUID NOT NULL REFERENCES auth.users(id),
  status VARCHAR(20) NOT NULL,          -- success, failed, skipped
  trigger_type VARCHAR(50),              -- event, scheduled, manual
  trigger_data JSONB,
  started_at TIMESTAMP NOT NULL,
  completed_at TIMESTAMP,
  duration_ms INT,
  error_context JSONB,                  -- Exception message, stack trace
  created_at TIMESTAMP NOT NULL
);

-- Table: automation_steps (created in Story 3.1)
CREATE TABLE automation_steps (
  id UUID PRIMARY KEY,
  execution_id UUID NOT NULL REFERENCES automation_executions(id),
  step_id VARCHAR(100) NOT NULL,
  status VARCHAR(20) NOT NULL,
  input JSONB NOT NULL,
  output JSONB,
  error_message TEXT,
  error_context JSONB,
  started_at TIMESTAMP NOT NULL,
  completed_at TIMESTAMP,
  duration_ms INT,
  created_at TIMESTAMP NOT NULL
);

-- Table: audit_logs (NEW)
CREATE TABLE audit_logs (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id),
  automation_id UUID REFERENCES automations(id),
  action VARCHAR(50) NOT NULL,          -- create, modify, enable, disable, delete, manual_retry
  old_values JSONB,                     -- What changed (before)
  new_values JSONB,                     -- What changed (after)
  ip_address INET,
  user_agent VARCHAR(500),
  created_at TIMESTAMP NOT NULL
);

-- Table: user_retention_policies (NEW)
CREATE TABLE user_retention_policies (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL UNIQUE REFERENCES auth.users(id),
  retention_days INT NOT NULL DEFAULT 90,
  archive_enabled BOOLEAN DEFAULT false,
  archive_bucket VARCHAR(255),          -- S3 bucket for archival
  updated_at TIMESTAMP NOT NULL
);

-- Indexes for performance
CREATE INDEX idx_automation_executions_user_automation_status
ON automation_executions(user_id, automation_id, status);

CREATE INDEX idx_automation_executions_created_at
ON automation_executions(created_at DESC);

CREATE INDEX idx_audit_logs_user_id_created_at
ON audit_logs(user_id, created_at DESC);

CREATE INDEX idx_audit_logs_automation_id
ON audit_logs(automation_id);
```

### Execution History Query

```typescript
interface ExecutionHistoryQuery {
  userId: string;
  automationId?: string;
  startDate?: Date;
  endDate?: Date;
  status?: 'success' | 'failed' | 'skipped';
  errorType?: string;
  searchTerm?: string;
  sortBy?: 'timestamp' | 'status' | 'duration';
  sortOrder?: 'asc' | 'desc';
  limit?: number;
  offset?: number;
}

async function queryExecutionHistory(query: ExecutionHistoryQuery) {
  let q = db('automation_executions')
    .where('user_id', query.userId)
    .select('*', 'automations.name as automation_name');

  if (query.automationId) {
    q = q.where('automation_id', query.automationId);
  }

  if (query.status) {
    q = q.where('status', query.status);
  }

  if (query.startDate && query.endDate) {
    q = q.whereBetween('created_at', [query.startDate, query.endDate]);
  }

  if (query.errorType) {
    // Search in error_context for specific error type
    q = q.where('error_context->error_type', query.errorType);
  }

  if (query.searchTerm) {
    q = q.where(function() {
      this.where('id', 'ilike', `%${query.searchTerm}%`)
        .orWhere('automations.name', 'ilike', `%${query.searchTerm}%`);
    });
  }

  // Pagination
  q = q
    .offset(query.offset || 0)
    .limit(query.limit || 50)
    .orderBy(`created_at`, query.sortOrder || 'desc');

  return q.leftJoin('automations', 'automations.id', 'automation_executions.automation_id');
}
```

### Audit Logging Middleware

```typescript
// Middleware for capturing user actions
async function auditLog(action: string, automationId?: string, changes?: any) {
  return db('audit_logs').insert({
    id: uuid(),
    user_id: req.user.id,
    automation_id: automationId,
    action,
    old_values: changes?.old || null,
    new_values: changes?.new || null,
    ip_address: req.ip,
    user_agent: req.get('user-agent'),
    created_at: new Date()
  });
}

// Usage in automation update endpoint
router.put('/automations/:id', async (req, res) => {
  const oldAutomation = await db('automations').where('id', req.params.id).first();

  // Update automation
  await db('automations').where('id', req.params.id).update(req.body);

  // Log the change
  await auditLog('modify_automation', req.params.id, {
    old: oldAutomation,
    new: req.body
  });

  res.json({ success: true });
});
```

### Retention Policy Cleanup Job

```typescript
async function cleanupOldExecutions() {
  // For each user, get their retention policy
  const policies = await db('user_retention_policies').select('*');

  for (const policy of policies) {
    const cutoffDate = new Date();
    cutoffDate.setDate(cutoffDate.getDate() - policy.retention_days);

    // Archive if enabled
    if (policy.archive_enabled) {
      const oldExecutions = await db('automation_executions')
        .where('user_id', policy.user_id)
        .where('created_at', '<', cutoffDate);

      // Export to S3
      await archiveToS3(oldExecutions, policy.archive_bucket, policy.user_id);
    }

    // Delete old records
    await db('automation_executions')
      .where('user_id', policy.user_id)
      .where('created_at', '<', cutoffDate)
      .delete();

    // Also delete related steps
    await db('automation_steps')
      .whereIn('execution_id', function() {
        this.select('id')
          .from('automation_executions')
          .where('user_id', policy.user_id)
          .where('created_at', '<', cutoffDate);
      })
      .delete();
  }
}

// Run nightly
schedule.scheduleJob('0 2 * * *', cleanupOldExecutions);  // 2 AM daily
```

## Architecture Integration

### Integration with Story 3.1: Execution Engine

This story implements the **observability layer** for the ExecutionService introduced in Story 3.1. [Source: docs/architecture/execution-engine-guide.md § Data Models, § Monitoring & Observability]

**How it extends Story 3.1:**

| Component | Story 3.1 | Story 3.5 | Purpose |
|-----------|-----------|-----------|---------|
| **AutomationExecution table** | Basic tracking (status, timings) | History + filtering + export | Enable debugging via UI |
| **AutomationSteps table** | Step-level execution data | Step timeline view | Visualize execution flow |
| **error_context (JSONB)** | Captured during execution | Displayed in detail panel | Show exception + stack trace |
| **Indexes** | User/automation/status/created_at | Same + query optimization | Enable fast querying at scale |

### Key Architectural Patterns

**1. Immutability & Audit Trail**
- Audit logs follow append-only pattern (similar to ExecutionContext immutability in 3.1)
- No mutation of historical data; new records only
- Enables compliance with regulations (GDPR, SOX, HIPAA)

**2. Performance Optimization**
- Composite indexes on `(user_id, automation_id, status)` for typical query patterns
- Separate index on `created_at DESC` for recency queries
- Pagination (50 rows/page) prevents memory exhaustion on large result sets

**3. Observability**
- Execution history queries use same patterns as Story 3.1 observability (see execution-engine-guide.md § Monitoring & Observability)
- Error context capture mirrors Story 3.1 error handling (message, stack, state snapshot)
- Background job pattern (retention cleanup) similar to async audit logging

### Security & Compliance Considerations

- **Audit logs immutable:** No edit/delete operations allowed (forensic integrity)
- **User isolation:** All queries filtered by `user_id` (multi-tenancy enforcement)
- **Retention policies:** User-configurable (90-2190 days) to comply with regulations
- **Data sanitization:** All exports scrubbed of sensitive fields before export
