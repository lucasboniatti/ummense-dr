## Technical Notes

### Rule Config Structure
```json
{
  "trigger": {
    "event_type": "task:created",
    "filters": [
      { "field": "task.priority", "operator": "equals", "value": "high" }
    ]
  },
  "conditions": [
    { "field": "task.status", "operator": "equals", "value": "todo" },
    { "field": "task.assigned_to", "operator": "not_equals", "value": null }
  ],
  "actions": [
    {
      "type": "send_notification",
      "params": {
        "message": "High priority task created",
        "priority": "high"
      }
    },
    {
      "type": "assign_tag",
      "params": { "tag_name": "urgent" }
    }
  ],
  "maxDepth": 3,
  "conditionLogic": "AND"
}
```

### Loop Detection
- Track rule execution chain: rule_1 → rule_2 → rule_3 (max 3)
- Block if: rule_X triggers itself directly or indirectly
- Example: Task:updated → rule_1 actions update task → task:updated event → rule_1 triggers again (BLOCKED)

### Rate Limiting
- Per-user: <100 rules max
- Per-rule: <1000 executions/day, checked at execution time
- Executions reset daily (UTC midnight)
- Increment execution_count on each run

### Atomic Execution
```
BEGIN TRANSACTION
  FOR each action IN rule.actions:
    IF action fails:
      ROLLBACK
      return error
  COMMIT
END
```

## Dev Notes

- Rule evaluation is synchronous (trigger event → check conditions → execute actions)
- Execution count is incremented after successful execution
- Loop detection uses stack trace in context, not database query
- Atomic transactions use PostgreSQL transaction support
