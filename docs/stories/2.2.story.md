## Story

As a **platform developer**, I want to **implement a rule-based automation engine** so that **users can create rules with triggers, conditions, and actions that execute atomically and reliably**.

---

## Acceptance Criteria

- [ ] **[CRITICAL - Architect]** Loop detection: Rules cannot trigger themselves (max 3 nested levels, block self-triggering)
- [ ] **[CRITICAL - Architect]** Rate limiting enforced: <100 rules per user, <1000 executions per rule per day
- [ ] **[CRITICAL - Architect]** Atomic action execution: All actions wrapped in transaction, all-or-nothing semantics
- [ ] **[CRITICAL - Architect]** Rule audit table created with full change history (create, update, delete, enable, disable)
- [ ] Rules table created with:
  - UUID primary key (id)
  - user_id (foreign key)
  - rule_name (VARCHAR 255, unique per user)
  - rule_version (INT for versioning)
  - config (JSONB: trigger, conditions[], actions[], maxDepth)
  - enabled (BOOLEAN)
  - execution_count (for rate limiting)
  - last_execution_at (TIMESTAMP)
  - created_at, updated_at, deleted_at
- [ ] Rule history table for audit trail with:
  - rule_id (foreign key)
  - changed_by (user who made change)
  - change_type ('created', 'updated', 'deleted', 'enabled', 'disabled')
  - old_config, new_config (JSONB before/after)
  - change_reason
- [ ] Rule evaluation engine with:
  - `evaluateCondition(condition, context)` function
  - Support for operators: equals, not_equals, greater_than, less_than, in, not_in, contains
  - JSON path support for nested field access (e.g., "task.status")
- [ ] Rule execution engine with:
  - `executeRule(ruleId, event, context)` function
  - Loop detection: Track execution depth, fail if >3 levels
  - Atomic transaction wrapper
  - Error handling and rollback on failure
- [ ] Action types supported:
  - `update_task`: Update task fields
  - `create_task`: Create new task from template
  - `send_webhook`: Call webhook endpoint
  - `send_notification`: In-app notification
  - `assign_tag`: Add tag to task
- [ ] Test coverage:
  - Loop detection prevents infinite chains
  - Rate limiting blocks >1000 exec/rule/day
  - Atomic execution: All actions succeed or all rollback
  - Rule history tracks all changes
  - Multiple conditions evaluated with AND/OR logic
  - Actions execute in dependency order
- [ ] **[HIGH - Architect]** Rule execution logging: All executions logged to automation_logs with status, duration, error
- [ ] **[MEDIUM - Architect]** Performance: Rule evaluation <100ms for typical rule with 5 conditions

## Technical Notes

### Rule Config Structure
```json
{
  "trigger": {
    "event_type": "task:created",
    "filters": [
      { "field": "task.priority", "operator": "equals", "value": "high" }
    ]
  },
  "conditions": [
    { "field": "task.status", "operator": "equals", "value": "todo" },
    { "field": "task.assigned_to", "operator": "not_equals", "value": null }
  ],
  "actions": [
    {
      "type": "send_notification",
      "params": {
        "message": "High priority task created",
        "priority": "high"
      }
    },
    {
      "type": "assign_tag",
      "params": { "tag_name": "urgent" }
    }
  ],
  "maxDepth": 3,
  "conditionLogic": "AND"
}
```

### Loop Detection
- Track rule execution chain: rule_1 → rule_2 → rule_3 (max 3)
- Block if: rule_X triggers itself directly or indirectly
- Example: Task:updated → rule_1 actions update task → task:updated event → rule_1 triggers again (BLOCKED)

### Rate Limiting
- Per-user: <100 rules max
- Per-rule: <1000 executions/day, checked at execution time
- Executions reset daily (UTC midnight)
- Increment execution_count on each run

### Atomic Execution
```
BEGIN TRANSACTION
  FOR each action IN rule.actions:
    IF action fails:
      ROLLBACK
      return error
  COMMIT
END
```

## Dev Notes

- Rule evaluation is synchronous (trigger event → check conditions → execute actions)
- Execution count is incremented after successful execution
- Loop detection uses stack trace in context, not database query
- Atomic transactions use PostgreSQL transaction support
