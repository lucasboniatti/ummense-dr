## Acceptance Criteria

- [ ] **[CRITICAL]** Execution engine supports sequential step execution with context passing between steps
  - Each step receives execution context (previous outputs, variables, automation config)
  - Step outputs stored and accessible to downstream steps
  - Execution order strictly sequential (no parallelization at this stage)
- [ ] **[CRITICAL]** Each step tracks input, output, status, start time, end time, and error details
  - ExecutionStep database model stores: step_id, execution_id, input, output, status, duration_ms, error_message
  - Status values: pending, running, success, failed
  - Timestamps in ISO-8601 format (UTC)
- [ ] **[CRITICAL]** Execution context (variables, previous step outputs) accessible to all steps
  - Context object passed as parameter to step execution function
  - Context includes: automation_id, trigger_data, user_id, step_outputs (previous steps)
  - All context immutable during step execution (prevent side effects)
- [ ] **[CRITICAL]** Backwards compatible: Wave 2 single-step automations execute unchanged
  - Wave 2 automations wrapped as single-step workflows internally
  - API endpoints for automation creation/update unchanged
  - Dashboard automation list shows existing automations without modification
  - No visible differences to end users
- [ ] **[HIGH]** Execution state stored in database for resumability and debugging
  - All executions persisted to automation_executions table
  - Step-by-step breakdown stored in automation_steps table
  - Execution state retrievable via API for debugging
  - Support query: "Show me the last 50 executions for automation X"
- [ ] **[HIGH]** 95%+ test coverage for execution engine
  - Unit tests for execution service: step execution, context passing, error handling
  - Integration tests: Wave 2 automations execute identically (output comparison)
  - Mock external API calls (no real HTTP requests in tests)
  - Test coverage metric: 95%+ lines of code tested
- [ ] **[MEDIUM]** Performance: No regression in single-step execution latency
  - P99 latency for single-step execution stable (within ±10% of Wave 2)
  - Execution context construction <10ms
  - Database inserts for execution tracking <50ms (async)
- [ ] **[MEDIUM]** Error handling: All exceptions captured with full context
  - Try/catch around step execution with detailed error logging
  - Error message, stack trace, and state snapshot stored in automation_steps.error_context
  - Failed executions don't cascade to other automations

## Technical Notes

### Execution Engine Architecture

```
automation execution starts
    ↓
load automation config (multi-step workflow)
    ↓
FOR EACH step in workflow:
  1. Build execution context (previous outputs, variables)
  2. Call step handler with context
  3. Capture output
  4. Store step record in database
  5. If error: capture error context, continue/fail based on config
    ↓
Store execution record (overall status, duration)
    ↓
Trigger webhook deliveries (if configured)
```

### Database Schema (ExecutionStep)

```sql
-- Table: automation_steps (NEW in Wave 3)
CREATE TABLE automation_steps (
  id UUID PRIMARY KEY,
  execution_id UUID NOT NULL REFERENCES automation_executions(id),
  step_id VARCHAR(100) NOT NULL,           -- From workflow definition
  status VARCHAR(20) NOT NULL,             -- pending, running, success, failed
  input JSONB NOT NULL,                    -- Step input (execution context)
  output JSONB,                            -- Step output (nullable if failed)
  error_message TEXT,
  error_context JSONB,                     -- Stack trace, state snapshot
  started_at TIMESTAMP NOT NULL,
  completed_at TIMESTAMP,
  duration_ms INT,
  created_at TIMESTAMP NOT NULL
);

-- Table: automation_executions (extend existing)
CREATE TABLE automation_executions (
  id UUID PRIMARY KEY,
  automation_id UUID NOT NULL REFERENCES automations(id),
  user_id UUID NOT NULL REFERENCES auth.users(id),
  status VARCHAR(20) NOT NULL,             -- pending, running, success, failed
  trigger_type VARCHAR(50),                -- event, scheduled, manual
  trigger_data JSONB,
  started_at TIMESTAMP NOT NULL,
  completed_at TIMESTAMP,
  duration_ms INT,
  error_context JSONB,
  created_at TIMESTAMP NOT NULL
);
```

### ExecutionContext TypeScript Definition

```typescript
interface ExecutionContext {
  automationId: string;
  executionId: string;
  userId: string;
  triggerType: string;
  triggerData: Record<string, unknown>;
  stepOutputs: Record<string, unknown>;    // {stepId: output}
  variables: Record<string, unknown>;      // User-defined variables
  timestamp: string;                       // ISO-8601
}

interface StepDefinition {
  id: string;
  type: string;                            // e.g., 'webhook', 'conditional', 'action'
  config: Record<string, unknown>;
  onError?: 'continue' | 'fail';
}

interface ExecutionResult {
  stepId: string;
  status: 'success' | 'failed';
  output?: Record<string, unknown>;
  errorMessage?: string;
  errorContext?: {
    message: string;
    stack: string;
    state: Record<string, unknown>;
  };
  durationMs: number;
}
```

### Backwards Compatibility Layer

```typescript
// Wrap Wave 2 single-step automations as step-based workflows
function convertLegacyAutomation(automation: Wave2Automation): ModernWorkflow {
  return {
    id: automation.id,
    version: 2,  // Mark as converted
    steps: [
      {
        id: 'legacy-step-1',
        type: automation.type,
        config: automation.config,
        onError: 'fail'
      }
    ]
  };
}

// At execution time, transparently convert output format
function executeLegacyStep(step, context) {
  const result = executeStep(step, context);

  // If legacy automation, merge step output into execution result
  // for backwards compatibility with existing code
  return result;
}
```

## Dev Notes

- Execution engine is the **foundation** for all Wave 3 features
- All subsequent stories (3.2-3.7) depend on this step-based execution model
- Wave 2 automations must continue to work **without any visible changes**
- Error context is critical for debugging (include stack traces and state snapshots)
- Async database inserts for execution tracking (don't block step execution)
- Use database transactions to ensure atomicity of multi-step executions
