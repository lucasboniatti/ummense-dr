## Story

**As a** TaskFlow developer,
**I want** the database schema created with all 4 core tables (flows, flow_columns, cards, tasks) in Supabase,
**so that** the backend can store and retrieve TaskFlow data from a real database instead of mocks.

---

## Acceptance Criteria

1. [x] Table `flows` created with columns: id (UUID PK), user_id (UUID FK → auth.users), name, description, created_at, updated_at
2. [x] Table `flow_columns` created with columns: id (UUID PK), flow_id (FK → flows CASCADE), name, color, order_index, created_at
3. [x] Table `cards` created with columns: id (UUID PK), flow_id (FK → flows CASCADE), column_id (FK → flow_columns CASCADE), title, description, order_index, created_by (FK → auth.users), created_at, updated_at
4. [x] Table `tasks` created with columns: id (UUID PK), card_id (FK → cards CASCADE), title, description, priority (CHECK P1/P2/P3), status (CHECK open/in_progress/done/blocked), due_date, assigned_to (FK), created_by (FK), created_at, updated_at
5. [x] All tables have RLS enabled with policies scoped to user ownership chain (flow.user_id)
6. [x] Indexes created for all foreign keys and common query patterns
7. [x] `updated_at` auto-update trigger on flows, cards, tasks tables
8. [x] Migration applies cleanly via `supabase db push`

## Dev Notes

### Architecture Reference
- **Full DDL:** `docs/architecture/taskflow-architecture.md` Section 2.2
- **ER Diagram:** `docs/architecture/taskflow-architecture.md` Section 2.1
- **Design Decisions:** `docs/architecture/taskflow-architecture.md` Section 2.3
- **Index Strategy:** `docs/architecture/taskflow-architecture.md` Section 2.4

### Key Design Decisions
[Source: docs/architecture/taskflow-architecture.md#2.3]
- PK type: UUID (Supabase default)
- No soft deletes — CASCADE deletes, backups via Supabase
- Timestamps: TIMESTAMPTZ (timezone-safe)
- `order_index`: INTEGER with gap strategy (10, 20, 30...)
- RLS: Single ownership chain flow → columns → cards → tasks
- Priority/Status: CHECK constraints, not lookup tables

### Supabase Client
[Source: packages/backend/src/lib/supabase.ts]
- `supabase` client uses service role key (bypasses RLS)
- `supabaseAuth` uses anon key (respects RLS)
- Backend uses service role + manual `user_id` filter (existing pattern from webhook.service.ts)

### Known Issue: Auth ID Type Mismatch
[Source: docs/architecture/taskflow-architecture.md#6.1]
- Current auth middleware uses `id: number` in JWT
- DB schema uses `UUID REFERENCES auth.users(id)`
- Recommendation: Use service role client + manual filter (matches existing pattern)

### Testing

- Test migration applies cleanly
- Test RLS policies: user A cannot see user B's flows
- Test CASCADE: deleting a flow removes all related columns, cards, tasks
- Test CHECK constraints: invalid priority/status rejected
