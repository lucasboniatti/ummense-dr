## Dev Notes

### Relevant Context

- O frontend já possui `EmptyState`, `PageLoader`, `Spinner`, `Toast`, `ToastContainer` e `ErrorBoundary`.
- O problema desta story é inconsistência de aplicação nas superfícies de produto, não ausência total de primitives.
- A padronização deve priorizar as rotas principais desta trilha em vez de tentar cobrir toda a aplicação em uma única passada.

### Loading Pattern

```tsx
// Pattern: Skeleton → Spinner → Content
function DataComponent() {
  const { data, isLoading, error } = useQuery();

  if (isLoading) {
    return <SkeletonCard />;
  }

  if (error) {
    return <ErrorBanner message={error.message} />;
  }

  if (!data || data.length === 0) {
    return <EmptyState title="Nenhum item" action="Criar novo" />;
  }

  return <DataList data={data} />;
}
```

### Error Pattern

```tsx
// Inline Error Banner
<ErrorBanner
  title="Erro ao carregar"
  message="Não foi possível carregar os dados."
  action={{ label: "Tentar novamente", onClick: refetch }}
/>
```

### Empty State Pattern

```tsx
<EmptyState
  icon={<Inbox className="w-12 h-12" />}
  title="Nenhuma tarefa encontrada"
  description="Crie uma nova tarefa para começar."
  action={{
    label: "Nova tarefa",
    onClick: () => openModal(),
  }}
/>
```

### Implementation Guidance

- Priorizar estados de loading/error/empty das superfícies operacionais mais visíveis.
- Reusar banners inline existentes antes de criar `ErrorBanner`.
- Documentar o padrão final para que as próximas stories reutilizem o mesmo contrato visual.

### Evidence Expectations

- Before/after dos estados de loading, error e empty nas rotas alvo.
- Evidência de toast em ação de sucesso e erro.
