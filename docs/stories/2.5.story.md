## Story

As a **user**, I want to **view automation execution metrics and logs on a dashboard** so that **I can monitor rule performance, debug failures, and optimize automation workflows**.

---

- [ ] **[CRITICAL - Architect]** Automation metrics table: Pre-aggregated metrics (hourly refresh)
  - user_id, metric_date, rules_count, webhooks_count, events_processed, automation_failures, avg_execution_time_ms
  - UNIQUE constraint on (user_id, metric_date)
  - Hourly refresh via scheduled job
- [ ] Automation dashboard page at `/dashboard/automations`:
  - KPI cards: Total rules, Total webhooks, Events processed (24h), Success rate, Avg execution time
  - Time series chart: Rule executions over last 7 days (success/failure breakdown)
  - Top failing rules: Table of rules with highest failure rate
  - Recent executions: Real-time list of last 50 executions with status, duration, error message
- [ ] **[HIGH - Architect]** Automation logs search interface:
  - Filter by rule_id, webhook_id, execution_status (success, failed, pending)
  - Filter by date range (last 24h, 7d, 30d, custom)
  - Search by error message or rule name
  - Sort by timestamp, duration, status
  - Export as CSV (max 10K rows)
- [ ] **[HIGH - Architect]** Execution detail view:
  - Rule configuration at time of execution
  - Conditions evaluated (true/false breakdown)
  - Actions executed (in order with status and duration)
  - Full error trace (if failed)

  - Success rate: Updates every 30 seconds
  - Failed execution alert: Notification when rule failure rate > 10% in last hour
- [ ] Alert configuration:
  - Failure rate threshold (default: 10% in last hour)
  - Max execution time threshold (default: 5s)
  - Notification method (in-app, email, webhook)
  - Enable/disable per rule
- [ ] Test coverage:

  - Filtering by status, date, search term works
  - Real-time updates via WebSocket
  - Alerts trigger on threshold breach
  - CSV export includes all matching records
- [ ] **[MEDIUM - Architect]** Performance: Dashboard loads <1s (uses pre-aggregated metrics, not real-time calculation)

## Technical Notes

### Pre-Aggregated Metrics Query
```sql
-- Dashboard KPIs - single aggregated query
SELECT
  COUNT(DISTINCT rule_id) as rules_count,
  COUNT(DISTINCT webhook_id) as webhooks_count,
  SUM(CASE WHEN created_at > now() - INTERVAL '24h' THEN 1 ELSE 0 END) as events_24h,
  SUM(CASE WHEN execution_status = 'success' THEN 1 ELSE 0 END)::float /
    COUNT(*) as success_rate,
  AVG(CASE WHEN duration_ms IS NOT NULL THEN duration_ms ELSE 0 END) as avg_duration_ms
FROM automation_metrics
WHERE user_id = $1
AND metric_date = CURRENT_DATE;
```

### Time Series Aggregation
```sql
-- 7-day execution breakdown (by status)
SELECT
  DATE(al.created_at) as execution_date,
  CASE WHEN al.execution_status = 'success' THEN 'success' ELSE 'failed' END as status,
  COUNT(*) as count
FROM automation_logs al
WHERE al.user_id = $1
AND al.created_at > now() - INTERVAL '7 days'
GROUP BY execution_date, status
ORDER BY execution_date DESC;
```

### Real-Time WebSocket Subscription
```javascript
// Subscribe to user's automation events
socket.on('automation:executed', (log) => {
  // Update live counter
  liveCounter += 1;

  // Update success rate (every 30s batch)
  batchLogs.push(log);
  if (batchTimer expired) {
    updateSuccessRate(batchLogs);
    batchLogs = [];
  }

  // Check alert thresholds
  if (checkFailureRateThreshold(log)) {
    showAlert('Rule failure rate exceeded 10%');
  }
});
```

### Alert Threshold Calculation
```sql
-- Check 1-hour failure rate for rule
SELECT
  COUNT(CASE WHEN execution_status = 'failed' THEN 1 END)::float / COUNT(*) as failure_rate
FROM automation_logs
WHERE rule_id = $1
AND created_at > now() - INTERVAL '1 hour'
HAVING COUNT(*) > 0;
```

### CSV Export Query
```sql
-- Export with all relevant fields
SELECT
  created_at,
  rule_id,
  webhook_id,
  execution_status,
  duration_ms,
  error_message,
  result
FROM automation_logs
WHERE user_id = $1
AND (filters...)
ORDER BY created_at DESC
LIMIT 10000;
```

## Dev Notes

- Metrics are pre-aggregated (no real-time calculation at dashboard load)
- Hourly refresh via `refresh_automation_metrics()` function (scheduled job)
- Real-time updates use WebSocket, not polling (lower latency, less server load)
- CSV export limited to 10K rows to prevent memory issues
- Alert thresholds are configurable per rule/webhook
- Dashboard uses aggregated metrics table for fast loads (<1s)
- Execution detail view loads from automation_logs (full detail, slower query)
