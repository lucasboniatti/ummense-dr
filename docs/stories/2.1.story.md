- [ ] **[CRITICAL - Architect]** Event deduplication implemented via UUID (event_id) to prevent duplicate processing
- [ ] **[CRITICAL - Architect]** Event versioning (event_version) supports schema evolution without breaking existing consumers
- [ ] **[CRITICAL - Architect]** WebSocket cleanup: Active connections timeout after 5 minutes of inactivity or 10 heartbeats missed
- [ ] **[CRITICAL - Architect]** Event log retention: 90-day soft delete with scheduled cleanup job
- [ ] Event log table created with:
  - UUID primary key (id)
  - user_id (foreign key to auth.users)
  - event_type (VARCHAR 50, format: "domain:action")
  - event_id (UUID for deduplication)
  - event_version (INT for versioning)
  - payload (JSONB for event data)
  - created_at, deleted_at (soft delete)
- [ ] Unique constraint on (user_id, event_id, event_type) for deduplication
- [ ] Index on (user_id, created_at DESC) for timeline queries
- [ ] Index on event_type for filtering
- [ ] Event emission service created with:
  - `emitEvent(userId, eventType, eventId, payload, version?)` function
  - UUID generation for eventId (if not provided)
  - Duplicate check before insert (returns idempotently if exists)
  - WebSocket broadcast to subscribed clients
- [ ] WebSocket EventEmitter integration:
  - Client subscribe: `socket.on('task:created', handler)`
  - Server emit: `emitter.emit('task:created', eventData)`
  - Automatic heartbeat (every 30s)
  - Connection cleanup on timeout or error
- [ ] Test coverage:
  - Event deduplication prevents duplicates

  - WebSocket receives events in real-time
  - Cleanup job removes events >90 days old
  - Multiple event types can be emitted concurrently

## Technical Notes

### Event Type Format
```
domain:action (e.g., "task:created", "task:updated", "task:completed", "tag:created", "automation:executed")
```

### Event Payload Example
```json
{
  "id": "task-123",
  "title": "New Task",
  "status": "todo",
  "userId": "user-456",
  "timestamp": "2025-03-01T10:00:00Z"
}
```

### Deduplication Strategy
- External systems send event_id (UUID) with each event
- Event_id + user_id + event_type = UNIQUE constraint
- App checks existence before inserting (idempotent)
- Returns success either way (duplicate = already processed)

### WebSocket Flow
```
Client subscribes → Server registers listener → Event emitted → Broadcast to subscribers
→ Client receives in real-time → Client updates UI
```

### Migration Reference
- Table: `public.event_logs` (created in migration 20250301000001)
- Soft delete policy: WHERE deleted_at IS NULL

---

## Dev Notes

- Event emission is synchronous (emit → broadcast immediately)
- Async processing for heavy operations (webhooks, automation rules) in separate job queue
- WebSocket heartbeat prevents stale connections from blocking clients
- Deduplication is database-level (UNIQUE constraint) + app-level (idempotent check)
