## Technical Notes

### Event Type Format
```
domain:action (e.g., "task:created", "task:updated", "task:completed", "tag:created", "automation:executed")
```

### Event Payload Example
```json
{
  "id": "task-123",
  "title": "New Task",
  "status": "todo",
  "userId": "user-456",
  "timestamp": "2025-03-01T10:00:00Z"
}
```

### Deduplication Strategy
- External systems send event_id (UUID) with each event
- Event_id + user_id + event_type = UNIQUE constraint
- App checks existence before inserting (idempotent)
- Returns success either way (duplicate = already processed)

### WebSocket Flow
```
Client subscribes → Server registers listener → Event emitted → Broadcast to subscribers
→ Client receives in real-time → Client updates UI
```

### Migration Reference
- Table: `public.event_logs` (created in migration 20250301000001)
- Soft delete policy: WHERE deleted_at IS NULL

---

## Dev Notes

- Event emission is synchronous (emit → broadcast immediately)
- Async processing for heavy operations (webhooks, automation rules) in separate job queue
- WebSocket heartbeat prevents stale connections from blocking clients
- Deduplication is database-level (UNIQUE constraint) + app-level (idempotent check)
