## Story

As a **user building complex automations**, I want to **create multi-step workflows with conditional branching logic (if/then/else)**, so that **I can execute different actions based on trigger data or intermediate execution results, enabling intelligent automation chains**.

---

## Acceptance Criteria

- [ ] **[CRITICAL]** Workflow execution engine supports multi-step sequences with conditional branches
  - Execution engine evaluates conditions (if/then/else logic)
  - Only active branches execute (dead branches skipped)
  - Context variables passed between steps
  - Step execution order respects dependencies

- [ ] **[HIGH]** Workflow Builder UI renders DAG (Directed Acyclic Graph) visualization
  - Visual representation of workflow steps and connections
  - Drag-and-drop step addition
  - Conditional branches displayed with "yes/no" paths
  - DAG renders without performance lag (<500ms for 50 steps)

- [ ] **[HIGH]** Conditional logic evaluator supports if/then/else expressions
  - Expressions: equality (==, !=), comparison (<, >, <=, >=)
  - Logical operators: AND, OR, NOT
  - Variables from trigger data and step outputs available
  - Invalid expressions rejected with clear error messages

- [ ] **[HIGH]** Workflow templates can be saved and loaded
  - Save current workflow as reusable template
  - Load template to create new automation
  - Template versioning supported
  - Templates listed in UI with preview

- [ ] **[MEDIUM]** Dry-run execution tests workflows without persisting results
  - Execute workflow with test data
  - Show execution trace (step-by-step output)
  - Display final output without side effects
  - Identify execution errors before running for real

- [ ] **[MEDIUM]** Existing Wave 2 automations migrate to new workflow model
  - Single-step automations convert to workflow format (backward compatible)
  - Migration is transparent (no user action required)
  - Wave 2 automations execute identically after migration
  - Rollback possible if issues found

- [ ] **[MEDIUM]** Execution history includes step-level breakdown
  - Each step logs: inputs, outputs, execution time, status
  - Branching decisions logged (which condition was true?)
  - Step-level error details captured
  - Query execution by workflow, date, status

## Dev Notes

### Workflow DAG Model & Architecture

**Foundation from Story 3.1 (Execution Engine):**
- ExecutionStep model already exists (supports multi-step sequences)
- Execution context passing implemented for variable sharing between steps
- Story 3.1 established the foundation for step sequencing

**DAG Model Design:**

A Directed Acyclic Graph (DAG) represents workflow structure:

```yaml
Workflow:
  id: string (automation_id)
  steps: Step[]
    - Step 1: Trigger (collect input data)
    - Step 2: Conditional Decision (if/then/else)
      - Branch A (true): Action Step
      - Branch B (false): Action Step
    - Step 3: Aggregate (merge results)

  # Execution Properties
  dependencies: TopologicalSort(steps)
  context_variables: Map<step_id, outputs>
```

**Key Architectural Decisions:**
1. **Acyclic Guarantee:** Validate no circular dependencies at save time
2. **Conditional Evaluation:** Single condition per branch node (no nested conditions v1)
3. **Context Isolation:** Each step sees parent context but cannot modify siblings
4. **Performance Constraint:** Render 50-step DAG in <500ms (React perf optimization needed)

### Database Schema Context

From Story 3.1's schema design (reference: docs/stories/3.1.story.md):
- `automations` table — main workflow definition
- `automation_steps` table — individual steps (type: trigger|action|decision)
- `automation_executions` table — execution records
- NEW for Story 3.4: `workflow_definitions` table (DAG metadata)
- NEW for Story 3.4: `workflow_step_executions` table (step-level audit)

### Conditional Logic Implementation

**Expression Format:** Simple binary logic
```typescript
// Valid expressions
"trigger.value == 'active'"
"previous_step.count > 100"
"previous_step.status == 'success' AND trigger.retry_count < 3"
"NOT (previous_step.error_code == 'TIMEOUT')"
```

**Operators Supported:**
- Comparison: `==`, `!=`, `<`, `>`, `<=`, `>=`
- Logical: `AND`, `OR`, `NOT`
- Variables: Available from trigger data + all previous step outputs

**Security Constraint:** No function calls, no arbitrary code execution (expression injection prevention critical)

### Backward Compatibility (Wave 2 Migration)

Story 3.3 automations are single-step (trigger → action).
Migration strategy: Auto-wrap in DAG with single branch (no conditions).

```typescript

// Wave 3 workflow { steps: [ Trigger, Action ], no conditions }
```
